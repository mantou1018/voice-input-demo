# Demo 实现方法论与 SOP

适用场景：

- 多状态 demo：同一页面会在准备中、输入中、处理中、确认中、错误态之间切换
- 设计还原 demo：同一页面需要对齐 Figma、截图或既有参考图
- 语音/AI/表单 demo：包含浏览器转写、字段抽取、手动补录、异步反馈
- 需要快速验证、快速给预览、但又不能反复返工的前端 demo

目标：

- 不再一边改代码一边猜需求
- 不再把同一页面做成多个长得不一样的页面
- 不再把视觉问题、状态问题、识别问题混在一起修
- 让后续类似 demo 有一套能直接照着执行的流程

## 一、核心方法论

以后做类似 demo，先把 4 个“真相”钉住，再进代码：

1. 产品真相：这次到底要改什么，不改什么
2. 状态真相：页面到底有几个状态，谁负责切换
3. 视觉真相：同一页面跨状态哪些元素绝对不能变
4. 链路真相：问题出在 UI、状态、转写、抽取、回填，还是环境

如果这 4 个真相里有任意一个不清楚，默认不要直接改主应用。

## 二、标准 SOP

### 0. 入场判断

先判断这次任务是不是“高返工风险 demo”。满足任意一条，就必须走完整 SOP：

- 同一页面有 3 个及以上状态
- 用户要前端还原，且会看细节一致性
- 有语音、ASR、AI 抽取、异步处理中任意一层
- 有“先别改，先给方案”或“我想象不出来”的信号
- 有手机端、安全区、弹层、底部操作区

如果只是静态小改字、小改颜色、小改间距，可以不走完整 SOP。

### 1. 需求冻结卡

进代码前先产出一张“需求冻结卡”。没有这张卡，不开始改主应用。

模板：

```md
## 需求冻结卡

- 目标：
- 这次要改的状态：
- 这次明确不改的状态：
- 必填字段：
- 是否允许优先级引导：
- 是否需要先给原型：
- 验收基准：
- 真实使用环境：
- 需要给用户的预览方式：
```

判定规则：

- 用户说“准备态和录音中保持现状”，就写进“不改的状态”
- 用户说“所有信息都是必填项”，就写进“必填字段”和“禁止优先级引导”
- 用户说“我想象不出来”，就写进“需要先给原型：是”
- 用户说“手机可打开”，就写进“真实使用环境：手机浏览器，不默认桌面 Chrome”

### 2. 状态表先行

任何多状态 demo，都先做状态表，再写组件。

模板：

| 状态 | 用户看到什么 | 允许操作 | 触发条件 | 下一状态 | 不变量 |
| --- | --- | --- | --- | --- | --- |
| prepare | 引导文案、空表单、开始入口 | 开始录音、关闭 | 首次进入 | recording | 标题区不漂、底栏位置不变 |
| recording | 输入反馈、语音提示 | 结束、重说 | 用户开始说话 | extracting/review/error | 表单骨架不变 |
| extracting | 处理中反馈 | 禁止重复提交 | 用户结束输入 | review/error | 顶部和底部骨架不变 |
| review | 识别结果、补充入口、确认入口 | 补充、手改、确认 | 处理完成 | recording/success | 表单区位置不变 |
| error | 错误提示、重试、手填 | 重试、手改 | 识别失败/权限失败 | recording/review | 同页骨架不变 |

执行规则：

- 状态命名要唯一，不要一层叫 `phase`，一层叫 `overlayMode`，再一层又拆 `showReview/showError/showExtracting`
- 页面应该有一个“状态真源头”，其他组件只消费，不重复推导
- 同一状态的标题、骨架、按钮区，不允许在多个组件各自决定

### 3. 视觉不变量清单

还原前先定义“跨状态不变量”，这一步比调 CSS 更重要。

模板：

```md
## 视觉不变量

- 页面宽高基准：
- 顶部返回按钮位置：
- 标题区 top / left / gap：
- 表单区整体宽度与起点：
- 聊天气泡区高度：
- 底部操作区 left / right / bottom：
- 主按钮高度 / 圆角 / 字号：
- 次按钮高度 / 圆角 / 字号：
- 安全区处理：
```

原则：

- 同一页面切状态，优先换内容，不要换版式
- 能共享的外层壳子，只做一个
- 字号、间距、圆角、阴影不要散落在各状态分支里
- 同类元素必须用统一 token 或统一 class，不要每个状态各写一份

### 4. 原型优先规则

以下情况不要直接改主应用，先做独立原型：

- 交互方案还没定
- 用户说“先别改，你先想方案”
- 用户说“我想象不出来”
- 同一功能有 2 个以上合理交互方案
- 改动可能波及 3 个以上状态

原型要求：

- 放在 `docs/` 下，做成可单独预览的 HTML
- 只验证交互和状态，不急着接真实数据
- 一次给 2 到 3 个方案，不要给 6 个半成品
- 原型定了，再回主应用

### 5. 分层实现顺序

推荐固定顺序，避免一上来就陷进细节。

1. 先收页面骨架  
目标：把标题区、表单区、聊天区、底栏区固定住

2. 再收状态切换  
目标：只保留一份状态真源头，确认每个状态能进能出

3. 再接字段回填和手动编辑  
目标：先保证数据流正确，再处理视觉细节

4. 再接异步链路  
目标：把浏览器转写、AI 抽取、回填各层串起来

5. 最后做文案、动效、阴影、细微间距  
目标：把抛光留到最后，不要边修逻辑边微调样式

禁止顺序：

- 还没定状态表就先写 4 个 if 分支
- 还没定骨架就先按状态分别还原
- 还没判层就直接改 AI prompt 或直接改 CSS

### 6. 一致性验收清单

验的不是“单页像不像”，而是“同页切状态有没有漂”。

每次至少检查这些状态：

- 初始态
- 输入中
- 处理中
- 确认态
- 错误态
- 手动编辑弹层

每个状态都检查：

- 标题文案是否正确
- 顶部返回/关闭按钮是否同位
- 表单起点、宽度、行高是否一致
- 聊天区高度和上下边界是否一致
- 底部按钮区是否同位
- 安全区是否正确
- 同类按钮的高度、圆角、字号是否一致
- 同类气泡的阴影、圆角、边距是否一致

判断标准：

- 如果你需要在 review 态单独微调 8px，recording 态单独微调 12px，通常说明骨架就已经不统一
- 如果一个状态改完，另一个状态要跟着补，通常说明样式没有共用

### 7. Bug 分层排查 SOP

不要看到现象就改最近的代码，先判层。

| 症状 | 先查哪层 | 不要先改什么 |
| --- | --- | --- |
| 同一页面不同状态长得不一样 | 页面骨架 / 视觉不变量 / 样式复用 | 不要先调单个状态的 magic number |
| 补充态和确认态来回切错 | 状态机 / 状态退出条件 | 不要先改文案 |
| 4 秒后自动发送不稳定 | transcript 更新 / timer 清理 / 状态切换 | 不要先改按钮文案 |
| 职位识别听错 | 浏览器 ASR 层 | 不要先怀疑 AI 抽取 |
| 字段识别对了但 UI 没回填 | 映射 / 回填 / picker 数据 | 不要先改语音层 |
| 手机能打开但不能用 | 浏览器支持 / 权限 / dev API 链路 | 不要先看桌面端 |

执行规则：

- 同一层连续试 2 次还没解决，就停止机械试错，回到根因判断
- 每次只改一层，改完立即验证，不要 UI、状态、抽取一起改

### 8. 测试与验证 SOP

测试必须分 3 层，不要只跑单测就结束。

第一层：纯逻辑测试

- 状态辅助函数
- 文案辅助函数
- 字段映射和回填
- 限制条件和边界值

第二层：主流程测试

至少覆盖这 5 条主路径：

1. 首次进入 -> 输入 -> 处理 -> 确认
2. 输入为空 -> 错误提示 -> 重试
3. 确认态 -> 继续补充 -> 返回确认态
4. 手动编辑字段 -> 确认可提交
5. 手机端关键浏览器下的权限或不支持分支

第三层：视觉验收

- 截图比对关键状态
- 检查跨状态骨架是否一致
- 检查手机安全区和底部按钮是否正确

通过标准：

- `build` 通过
- 逻辑测试通过
- 至少一轮真实页面验收完成
- 能提供可访问预览或对应文件链接

### 9. 交付 SOP

每次交付都带这 4 类信息：

```md
- 改了什么
- 为什么这样改
- 去哪里看
- 怎么验证
```

如果是 demo：

- 给预览链接或文件链接
- 说明当前验证过哪些状态
- 明确还没验证的风险点

## 四、常见反模式与对应对策

| 反模式 | 后果 | 对策 |
| --- | --- | --- |
| 需求没冻结就开改 | 越改越多，返工频繁 | 先写需求冻结卡 |
| 没有状态表 | 不同状态互相污染 | 先做状态表，再进组件 |
| 同页多态各画一遍 | 前后一致性差 | 先做共享骨架，再换内容 |
| 样式分散在 if 分支里 | 一改一串漂 | 提前抽视觉 token 和共享 class |
| 看到识别错就先改 AI | 错层修 bug | 先判链路层级 |
| 只跑单测不看真机 | 实际体验不稳 | 增加主流程和视觉验收 |
| 一边调逻辑一边抠像素 | 成本很高 | 先收状态，再做视觉抛光 |

## 五、执行门槛

以后类似 demo，只要满足下面任意 2 条，就默认启用本 SOP：

- 有多状态切换
- 需要高保真还原
- 需要手机端验收
- 有语音或 AI 链路
- 用户已经对一致性或 bug 成本表达不满

## 六、这份 SOP 在本项目里的使用方式

建议组合使用：

- 本文档：用于控制方法、流程、验收
- [figma-frontend-fidelity.md](./figma-frontend-fidelity.md)：用于做具体页面和状态的还原核对

推荐执行顺序：

1. 先写需求冻结卡
2. 再写状态表和视觉不变量
3. 如果交互未定，先做 `docs/` 原型
4. 原型确认后再进主应用
5. 实现时按“骨架 -> 状态 -> 数据 -> 异步 -> 抛光”的顺序推进
6. 最后按一致性清单和分层排查 SOP 验收
